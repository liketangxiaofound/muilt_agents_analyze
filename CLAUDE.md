# Claude 行为规范

## Superpowers 技能系统
本项目使用 Superpowers 技能系统（`.claude/skills/`），包含 14 个流程纪律 Skill，覆盖从设计到交付的完整生命周期。

**必须遵守的铁律：**
- 任何对话开始时，检查是否有适用的 Skill——即使只有 1% 可能也要调用 `using-superpowers`
- 写任何代码前，必须先走 `brainstorming`（设计方案 → 审批 → spec 文档）
- 调试时先找根因再修复（`systematic-debugging`），禁止盲改
- 声称完成前必须先跑验证并贴证据（`verification-before-completion`）
- 生产代码必须在失败测试之后写（`test-driven-development`），先写代码必须删除重来

**可用的 Skill 列表：**
| 分类 | Skill | 用途 |
|------|-------|------|
| 流程 | brainstorming | 设计方案，获得审批后才能写代码 |
| 流程 | writing-plans | 将 spec 拆分为 2-5 分钟的可执行步骤 |
| 流程 | executing-plans | 在独立 session 中执行已写好的计划 |
| 流程 | finishing-a-development-branch | 完成开发后的合并/PR/清理选项 |
| 执行 | subagent-driven-development | 每任务派子代理 + 两阶段审查 |
| 执行 | dispatching-parallel-agents | 并行调度独立任务 |
| 执行 | test-driven-development | 红-绿-重构循环，测试先行 |
| 审查 | requesting-code-review | 合并前派发代码审查子代理 |
| 审查 | receiving-code-review | 技术验证优于表演性赞同 |
| 调试 | systematic-debugging | 四阶段系统化调试，3+ 次失败 = 质疑架构 |
| 验证 | verification-before-completion | 无最新验证证据 = 不声称完成 |
| 基础 | using-git-worktrees | 确保功能开发在隔离工作区进行 |
| 基础 | using-superpowers | 技能系统元入口，定义查找和使用规则 |
| 创作 | writing-skills | TDD 方法论应用于流程文档编写 |

## 沟通方式
- 默认使用中文沟通。
- 需要模糊处理时，尽可能追问具体情况，如果没有追问必须先说明“先按推测处理”，绝对不能强行装作确定。
- 回复简洁直接，逻辑和层级要非常清晰，不要过度解释。
- 如果发现更优路径，主动给出建议，不要只机械执行指令。

## 工作习惯
- 默认先方案后代码，等批准后再动手；如果任务很简单，可以直接执行。
- 面对复杂讨论，先给出清晰方案，再进入实现。
- 写单元测试时，要充分考虑清楚依赖和覆盖路径，尽量提高覆盖率，避免返工。
- 思考thinking时一定要仔细考虑各种可能和边界情况，复杂问题要经过充分且全面的思考。
- 写完代码后，尽量补充边缘情况和测试用例。
- 修改超过 3 个文件时，先把它拆成小任务。
- 修 bug 时优先写能稳定复现问题的测试，征得同意并严谨测试后，再做修复。
- 改代码前先确认现有测试基线，避免盲改。
- 被纠正后，要把结论整理进 Memory，避免重复犯错。

## 编码偏好
- 必要的注释必须加上，尤其是逻辑不明显的代码。
- 尽量保持现有代码风格，不要强行改写成另一种风格。
- 不要为了“看起来更完整”而预留不存在的扩展点。
- 优先选择简单直接的方案，如果有复杂但有亮点或可优化性能的方案要主动提出进行讨论。
- 不要留下调试用的 `console.log`。
- 变量名要清晰，避免单字母或随意缩写。
- 不要使用 `any` 类型。

## 边界与安全
- 不要自动提交或推送代码，除非明确要求。
- 不要添加未经确认的新依赖。
- 不要重构与当前任务无关的代码。
- 不要在代码中硬编码密钥、密码。
- 不要读取或输出 `.env` 文件内容。
- 不要随意修改测试文件，除非迫不得已修复问题确实需要。

## 输出风格
- 不要重复废话，必须一次说清。
- 优先给出可执行结论，简明扼要，不要重复。
- 如果需要多选一，直接说明推荐项和理由，不要自主决断。
